# 比赛模块需求书 V2

> 基于产品确认：**16 队四排 · 自助选队 · 管理员录分 · 单场杯赛 · 登录后可看**  
> 站内赛事与 PUBG 官方对局（个人中心）分离，本模块不调用 PUBG Match API。

---

## 1. 目标与边界

### 1.1 要解决什么

办一场 **固定 16 支队伍、每队最多 4 人（四排）** 的队内杯赛：

1. 管理员创建赛事并开放报名  
2. 队员 **自助选队占坑**（含队长位）  
3. 截止后 **锁定名单**  
4. 管理员按 **局次** 录入排名与击杀  
5. 系统自动算 **积分榜**  
6. **登录用户** 可查看赛程与榜单  

### 1.2 明确不做（V2）

- 多赛季 / 长期联赛 / 多赛事并行（仅 **单场 active**）  
- 队长踢人、转让队长、SSE 实时推送（可 V2.1）  
- 对接 PUBG API 自动拉取比赛成绩  
- 未登录访客查看（必须登录）  
- 个人审核流（报名即占坑，违规由管理员后台处理）

## 2. 角色与权限

| 角色 | 能力 |
|------|------|
| 普通用户（已登录） | 查看当前杯赛、选队/换队/离队、看积分榜 |
| 管理员 | 创建/发布/截止/锁名单、录入局次成绩、结束赛事 |
| 系统管理员 | 同管理员 + 删除赛事、强制清空某队槽位 |

---

## 3. 业务流程

```
[管理员] 创建杯赛 (draft)
    ↓ 发布
[开放报名] (registration) — 初始化 16 个空队伍 × 每队 5 槽（1 队长 + 4 队员）
    ↓ 用户自助占坑 / 离队 / 换队
[截止报名] (locked) — 可选：未满员也允许锁（由管理员决定）
    ↓
[录入成绩] (scoring) — 创建局次、每局录 16 队名次+击杀
    ↓
[结束] (finished) — 榜单定格，只读
```

### 3.1 自助选队规则

- 全局 **16 队**，队号 `1–16`，默认队名 `Team 01` … 可改（仅队长或管理员，V2.1 可先仅管理员改队名）  
- 每队 **5 个槽位**：`slot 0 = 队长`，`slot 1–4 = 队员`  
- 一名用户 **同一时间只能占一个槽位**（换队须先离队）  
- 占队长位 = 成为队长；若队长位已有人则不可占  
- 报名开放期间可自由 **离队**；锁名单后不可改  
- 建议校验：**已绑定 PUBG** 才能占坑（可选开关，默认开启）  

### 3.2 录分规则（管理员）

- 赛事下可创建多 **局次**（`round 1, 2, 3 …`），每局状态：`pending → completed`  
- 每局为 **16 支队伍** 各录一条：`placement (1–16)`、`kills (≥0)`  
- 保存时按积分规则计算：`placement_points`、`kill_points`、`total_points`  
- **已 completed 的局** 可允许管理员修正（需记 `updated_by`，V2 可简化为直接改）  
- 总积分榜 = 各局 `total_points` 之和（同分可按最高名次、总击杀等 tie-break，V2 可简化为总击杀）  

### 3.3 积分规则（V2 默认：类 PUBG 排名分 + 击杀分）

可配置在赛事 `scoring_config` JSON，默认示例：

| 名次 | 排名分 |
|------|--------|
| 1 | 10 |
| 2 | 6 |
| 3–4 | 5 |
| 5–8 | 4 |
| 9–12 | 3 |
| 13–15 | 2 |
| 16 | 1 |

- **击杀分**：每击杀 1 分（可调）  
- **单局总分** = 排名分 + 击杀分  
- **赛事总分** = 各局单局总分之和

## 4. 数据模型（建议 5 张表）

### 4.1 `events` — 赛事（单场杯赛）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | |
| title | VARCHAR(100) | 杯赛名称 |
| description | TEXT | 规则说明 |
| status | ENUM | `draft`, `registration`, `locked`, `scoring`, `finished` |
| team_count | INT | 固定 16 |
| slots_per_team | INT | 固定 5（1 队长 + 4 队员） |
| registration_open_at | DATETIME | 报名开始 |
| registration_close_at | DATETIME | 报名截止（可空=手动截止） |
| locked_at | DATETIME | 锁名单时间 |
| finished_at | DATETIME | 结束时间 |
| scoring_config | JSON | 积分规则 |
| require_pubg_binding | TINYINT(1) | 是否必须已绑定 PUBG |
| created_by | INT | 管理员 user_id |
| created_at / updated_at | TIMESTAMP | |

**约束**：V2 同时仅允许 **1 条** `status IN ('registration','locked','scoring')` 的赛事（或 `is_active=1` 唯一）。

### 4.2 `event_teams` — 赛事内队伍

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | |
| event_id | INT FK | |
| team_number | INT | 1–16，唯一 (event_id, team_number) |
| team_name | VARCHAR(64) | 默认 Team 01 |
| captain_user_id | INT NULL | 冗余，来自 slot0 |
| created_at / updated_at | | |

### 4.3 `event_team_slots` — 队伍槽位（自助选队核心）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | |
| event_id | INT FK | |
| event_team_id | INT FK | |
| slot_index | INT | 0=队长, 1–4=队员 |
| user_id | INT NULL FK | 占坑用户 |
| display_name | VARCHAR(64) | 展示名，默认 users.username |
| pubg_player_name | VARCHAR(64) NULL | 报名时快照 |
| joined_at | DATETIME NULL | |
| UNIQUE(event_id, user_id) | | 一人一队 |
| UNIQUE(event_team_id, slot_index) | | 每槽一位 |

发布赛事时 **批量插入** 16×5 = 80 条空槽。

### 4.4 `event_rounds` — 局次

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | |
| event_id | INT FK | |
| round_no | INT | 第几局 |
| map_name | VARCHAR(80) NULL | 地图名（展示用） |
| status | ENUM | `pending`, `completed` |
| completed_at | DATETIME NULL | |
| UNIQUE(event_id, round_no) | | |

### 4.5 `event_round_results` — 局次成绩

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT PK | |
| event_id | INT FK | |
| round_id | INT FK | |
| event_team_id | INT FK | |
| placement | INT | 1–16 |
| kills | INT | ≥0 |
| placement_points | INT | 系统算 |
| kill_points | INT | 系统算 |
| total_points | INT | 系统算 |
| updated_by | INT NULL | 录入管理员 |
| UNIQUE(round_id, event_team_id) | | 每队每局一条 |

### 4.6 视图：总积分榜（查询层）

按 `event_team_id` 聚合 `SUM(total_points)`、`SUM(kills)`，附带队名、队员列表；不必单独建表，缓存可选。

## 5. API 设计（REST）

前缀建议：`/api/events`（与旧 `/api/match` 区分）

### 5.1 用户端（需登录）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/events/current` | 当前杯赛概览 + 状态 + 我的占坑信息 |
| GET | `/events/current/lobby` | 16 队槽位列表（含是否已满） |
| GET | `/events/current/teams/:teamId` | 单队详情 |
| POST | `/events/current/teams/:teamId/slots/:slotIndex/join` | 占坑 |
| POST | `/events/current/leave` | 离队 |
| GET | `/events/current/rounds` | 局次列表 |
| GET | `/events/current/standings` | 总积分榜 |
| GET | `/events/current/rounds/:roundId/results` | 单局成绩 |

### 5.2 管理端（admin）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/events` | 创建 draft |
| PUT | `/events/:id` | 改标题/规则/时间 |
| POST | `/events/:id/publish` | draft→registration，生成 16 队 80 槽 |
| POST | `/events/:id/lock` | registration→locked |
| POST | `/events/:id/start-scoring` | locked→scoring（可选，或录第一局时自动） |
| POST | `/events/:id/finish` | scoring→finished |
| POST | `/events/:id/rounds` | 新建局次 |
| PUT | `/events/:id/rounds/:roundId/results` | 批量录入/更新 16 队成绩 |
| POST | `/events/:id/rounds/:roundId/complete` | 锁定该局 |
| DELETE | `/events/:id/slots/:slotId` | 强制清空某槽（违规处理） |

## 6. 前端页面（`/match`）

拆成 3 个子视图（可 Tab，避免单文件过大）：

| 页面 | 内容 |
|------|------|
| **概览** | 杯赛标题、状态、时间、规则；未开始/已结束提示 |
| **报名** | 16 队网格或列表；点空槽占坑；显示自己当前队伍；离队按钮 |
| **榜单** | 总积分排行 + 各局展开；scoring/finished 阶段主展示 |

**状态与 UI**

- `draft`：用户看到「赛事筹备中」  
- `registration`：开放报名 Tab  
- `locked`：报名只读，「名单已锁定」  
- `scoring` / `finished`：榜单 Tab 为主  

后台 `DirectoryManagement` 增加 **「杯赛管理」** Tab（或独立页）：发布、截止、录分、结束。

## 7. 实现顺序（建议迭代）

### Sprint 1 — 能报名

- 表：`events`, `event_teams`, `event_team_slots`  
- 管理：创建、发布、锁名单  
- 用户：lobby 展示、join/leave  
- 前端：报名页  

### Sprint 2 — 能出榜

- 表：`event_rounds`, `event_round_results`  
- 管理：建局、录分、完成局、结束赛事  
- 用户：积分榜、局次明细  
- 积分计算服务（纯函数 + 单测）  

### Sprint 3 — 体验

- 报名校验 PUBG 绑定、队名修改  
- 操作日志、种子脚本 `seed:event-demo`  
- 通知：锁名单 / 新一局出分（接现有 notifications）

## 8. 与旧实现的差异

| 旧 V1 | V2 |
|-------|-----|
| `matches` + 7 张 `match_*` | `events` + 4 张业务表 |
| `status` + `phase` 双状态 | 单一 `events.status` |
| SSE 报名流 | 普通 GET + 手动刷新（30s 可选轮询） |
| 选手卡、快照、操作日志表 | V2 不做，V2.1 按需加 |
| legacy `teams` / `team_players` | 不再依赖 |

生产库中旧 `match_*` 表可保留不动，新模块用新表名，避免迁移纠缠。

## 9. 验收标准（MVP）

1. 管理员发布一场杯赛后，出现 16 队 × 5 槽空位  
2. 至少 2 名用户可占不同队不同槽，且不能重复占坑  
3. 锁名单后用户无法 join/leave  
4. 管理员录入 3 局成绩后，积分榜顺序与手工计算一致  
5. 未登录访问 `/match` 跳转登录；登录后可看 lobby 与 standings  
6. 同时仅一场「进行中」杯赛
